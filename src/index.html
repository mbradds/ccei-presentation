Where have we used Highcharts so far?

Is Highcharts better than Tableau or PowerBI?

No... But we need to think about what we are trying to accomplish. Organizations such as ours have two completely separate functions, and our choice of technology needs to reflect this.

Think of it this way. If you want to build a house, how many tools would you bring to the worksite? What would happen if you showed up with only a hammer in your toolbox? This is a classic technology pitfall of most large organizations!

Are we interested in creating high quality web based dashboards and data products for external audiences? If so, then take a look at some of the reasons to switch! If not, then we can probably end the presentation here...

Reasons to switch

1. Latency

Latency is the time difference between when a website user requests something, and when they get what they want.

Try clicking through some of the filter options in this Tableau dashboard. Notice the time delay? Its not the end of the world, but users really dont like this, and there is nothing that can be done about it. Tableau and PowerBI were not designed to play well on the web.

Tableau and PowerBI are rendered through something called an iframe, which is kind of like looking through a large tunnel into another computer. User interactions need to pass through this tunnel and back again in order for the user too see what they want. This is kind of a hack/band aid solution for the web, but its the only way for Tableau and PowerBI to work.

As we have seen, Highcharts (and other web/JavaScript) based web frameworks dont have this latency issue, because all files originate on the orginizations web server, and the vizualization runs inside the users web browser, eliminating the need for an iframe.

Note: This is only an issue on the web! When creating Tableau/PowerBI dashboards on your computer, you wont notice this issue, because the vizualization is running on your local computer, instead of a server.

2. Responsive web design

Highcharts, and other web based frameworks work well on different screen sizes out of the box.

Mobile is no longer an afterthought. >50% of all web traffic comes from mobile phones.

3. Network transfer size & SEO
  • CCEI front page currently transfers 5.5MB uncached. This could be cut by 56% by moving to Highcharts.
  • CCEI front page currently requests 67 domains. This could be cut to 27 (59% improvement) by moving to Highcharts.

Improvements in both page size and total number of network requests could go a long way to improving the user experience, and improve SEO. Google doesnt like large, slow pages.

4. Reliability

Tableau and PowerBI depend on large amounts of external resources and servers outside of our contol. Tableau public outages and problems have burned us bad in the past...

Our Developers are currently working on resolving a Tableau Public issue. It affects certain authoring actions (web and Desktop) which involve loading or publishing a Tableau Public workbook. Additional updates will be published to the Trust: https://t.co/DShKM0T4aF Thanks! -CT

— Tableau Support Team (@TableauSupport) March 25, 2021

When set up properly, Highcharts vizualizations dont depend on external resources/servers, and can only fail when the organizations web server is down.

5. Customization

Tableau and PowerBI are intended to be used as internal business intelligence tools for management decision making and analysis. What happens when we want to build a visualiztion with functionality that isnt supported? What happends when our external users/audience request products that Tableau/PowerBI cant handle?

There are several examples of functionality on the previous slide that cant be re-created with Tableau or PowerBI. When moving to the Highcharts/JavaScript ecosytem, there are no limits, because there is always going to be a right tool for the job.

6. Data security, governance, and open source

What is Tableau & PowerBI doing with our data, and more importantly, what are they doing with our users data? The fact that all user interactions pass through Tableau/PowerBI servers means that they can do whatever they want.

Highcharts is open source. This means that its (almost) impossible for unsavory code to make its way in. Also, all the code we write on top of Highcharts is open source, so our clients can confirm that nothing shady is going on.

Our audience (the general public) is starting to care more about this. What have we seen so far where this is critical? Users dont like giving their exact location to websites, and probably not to government websites. We can prove that we are not saving this information!

7. JavaScript Ecosystem

Moving to Highcharts unlocks the benefits that come with the JavaScript ecosystem/community. Highcharts plays nicely with the official Government of Canada styles (web experience toolkit), helping to create a common look and feel.

Highcharts plays nicely with other open source (and fully free) software, such as leaflet for advanced mapping. This ensures that you can use the absolute best tool for the job!

What happens when something better than Highcharts comes along? Switching from one web based charting framework to another is much easier than the initial shift from PowerBI to Highcharts.

The JavaScript ecosystem allows for much easier sharing of work, and re-using old work compared to Tableau or PowerBI. NPM (node package manager), allows me to download and use JavaScript packages (such as Highcharts), but I can also create my own NPM packages that others can use.

8. Price $

I dont know the exact price tag for Tableau or PowerBI, but the difference compared to Highcharts may be significant. The difference in price must be weighed against the const of adopting a new technology, re-training, etc.

Highcharts + Highmaps license costs 722 USD. This is a one time payment, with one year of product support.

Reasons not to switch

This is a completely different way of doing things. It requires different skills compared to Tableau/PowerBI. Introducing a new technology also introduces alot of problems. We are going through alot of these problems right now.

This is not a discussion about replacing, removing, or phasing out Tableau/PowerBI. Tableau/PowerBI are irreplaceable, but they are not always the best tool for the job.

This is not an obvious decision, and it shouldn't be rushed into. However the case for this kind of technology shift will only get stronger in the future. I've categorized the reasons not to switch into three categories; Challenges, Risks, and Communication. Every organization is different, but these are some of the biggest issues i've faced.

Challenges
  • Skills - This is harder to learn than Tableau/PowerBI. Hiring experienced data viz developers is expensive, and they are hard to come by.
  • Beating Tableau - There is a reason Tableau/PowerBI are so successful. Their product is really good. Its hard to create polished products that are not just better than a tableau dashboard, but also good enough to turn heads in the organization. Tableau is so engrained in everything that we do, so things that are not done in tableau are challenged (rightfully so). The products I develop cant just be a little bit better than Tableau/PowerBI, they need to be so far ahead that a shift is worthwhile.
  • New processes - Delivering results with this new technology requires much closer collaboration with our web team. Even when I do "all the work", this group still needs to do alot to get things over the finish line. The only problem I've had on this front is we dont have a big enough web team!
Risks
  • Splitting the market analyst/data role - I'm not sure how things are done at CCEI, but at the CER its pretty common for the typical "market analyst" to have a heavy hand in the data/design world as well. Having the best of both worlds means we can be productive and move quickly.
  • Me
Communication
  • Buy In - Its difficult to convince people that a technology change is needed, when everything appears to be going just fine. This is a big one. Its not enough to just let the work speak for itself, If I cant communicate the benefits and potential that a new technology provides, then it wont happen.
Mitigations
  • Products like Highcharts didnt exist the way they do even five years ago. The JavaScript/open source/web community is progressing so fast, that making this shift is becomming easier and cheaper every year.

Why Highcharts specifically?

There is no reason. However Highcharts has some advantages over other open source technologies that moved the needle for us.

  • ApexCharts.js
  • chartist.js
  • plotly
  • Countless others
